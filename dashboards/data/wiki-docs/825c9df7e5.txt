# SharePoint Migration Guide

Amazon-wide Quip deprecation is moving from roadmap to real for marketing teams. The ABBizOps Intake example from April 17 shows the pattern: Quip doc replaced by SharePoint Online Excel, cross-functional partners rerouted. This guide documents the migration approach we used for the paid search wiki — 462 files, 207 docx files published, zero losses — so other teams can replicate rather than improvise.

## What migrated

The paid search wiki started as a mix of markdown files in `~/shared/wiki/` and referenced Quip documents. The target state was: markdown as the editable source of truth, SharePoint `Documents/Artifacts/wiki-sync/` as the published work-product copy, Kiro Pipeline view as the discovery layer. Quip was not a target.

The migration moved 462 files. Of those, 207 were published to SharePoint as docx (status REVIEW or FINAL). 255 were filtered out (DRAFT status or archived). Zero files were lost. Every migration action is logged, so any file can be traced from Quip URL through markdown edit history through SharePoint path.

## The three-layer architecture

The working model has three layers that must stay in sync. The **local markdown layer** lives at `~/shared/wiki/agent-created/` and is where every article is written and edited. The **SharePoint docx layer** at `Documents/Artifacts/wiki-sync/` is the published version humans consume. The **Kiro Pipeline view** is the discovery layer, built by `build-wiki-index.py` and served from `shared/dashboards/wiki-search.html`.

When a file changes in the local markdown layer, the sync tool generates a docx and uploads it to SharePoint. The Pipeline view re-indexes and shows the updated status. Cross-device access works through SharePoint; collaborative discovery works through the Pipeline view; authoring works locally.

## Migration approach

Four steps carried us from initial state to clean migration. First, batch-add frontmatter to every file with relevant defaults per directory (callouts FINAL, agent-created REVIEW, archive archived). The remediation script processed all 462 files. Second, fix the parse order — HTML comment before frontmatter block was breaking python-frontmatter; comment-after-frontmatter became the canonical order. Third, stage locally via the sync tool's converter, then upload via SharePoint MCP (the tool's default output path targeted a Windows OneDrive folder that doesn't exist on DevSpaces). Fourth, wipe-and-resync after filename cleanups rather than renaming individual SharePoint files.

## What to avoid

Avoid editing docx directly on SharePoint. Any edit there is lost on the next sync. Avoid absolute URLs in article content — they break when paths change. Avoid letting status values drift; DRAFT files should not appear in published Artifacts. Avoid skipping the frontmatter layer; without it, the Pipeline view cannot categorize the article.

## Replicating for another team

Other teams facing Quip deprecation should follow the same architecture. Four prerequisites: a markdown authoring environment (DevSpaces, GitHub, or equivalent), SharePoint MCP access, a sync tool that converts markdown to docx (we used `shared/tools/sharepoint-sync/cli.py`), and a discovery layer if the team wants search. If the team does not need search, they can skip the third layer and link directly to SharePoint.
