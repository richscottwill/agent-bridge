# Requirements Document: Intelligence Action Links

## Introduction

Add per-row "action" links to the Actionable Intelligence table in Command Center that open the right tool for each item type — Outlook compose for Communicate items, Asana task creation for Delegate items, and the relevant SharePoint artifact or doc for Differentiate items. Today, the Intelligence table shows what to do (Communicate / Delegate / Differentiate) with a dismiss button, but no "do it" button. Every item requires Richard to context-switch to another tool, remember where to go, and manually transfer the context. This spec closes the gap between "I see what to do" and "I'm doing it" by generating a single action URL per row that opens the right tool with context pre-filled.

**Origin**: `.kiro/specs/dashboard-learnings-roadmap/design.md` — Item #13 (Inline action links from Intelligence table). Ranked #13 of 13 roadmap items. Classification: `IMPROVE`, effort M (1–3 days), leverage High. Sources: babel-system asset URL field (link each asset to its live page), advertising-hub Buddy orchestrator routing (click → route to specialist agent).

**Five Levels alignment**: L1 (Sharpen Yourself) — reduces the gap between seeing an action item and executing it. Every second spent figuring out "where do I go to do this?" is a second not spent on the hard thing. Directly supports the L1 key metric (consecutive weeks shipped) by removing friction from the action-taking step of the morning routine.

**Soul principles**:
- Reduce decisions not options — an "action" link per row that opens the right tool (Outlook for Communicate, Asana for Delegate, relevant doc for Differentiate) reduces the "where do I go to do this?" decision to zero. The options don't change; the decision about which tool to open disappears.
- Structural over cosmetic — this isn't a visual redesign of the Intelligence table. It's a structural change: each row gains a pre-computed action URL that encodes the context (recipient, subject, task title, artifact path). The table's appearance barely changes; its utility jumps.
- Invisible over visible — the URL generation happens during the data refresh pipeline, not at click time. Richard doesn't see the plumbing — he sees a button that opens the right thing.

## Glossary

- **Actionable Intelligence table**: A section in Command Center that displays items categorized as Communicate, Delegate, or Differentiate — each representing a specific action Richard should take. Currently has a dismiss button per row but no action button.
- **Communicate item**: An Intelligence table row where the recommended action is to send a message — typically an email via Outlook or a Slack message to a specific person about a specific topic.
- **Delegate item**: An Intelligence table row where the recommended action is to assign a task to someone — typically creating an Asana task with context pre-filled (title, description, assignee).
- **Differentiate item**: An Intelligence table row where the recommended action is to review or create a strategic artifact — typically opening a SharePoint document, wiki article, or dashboard view relevant to the item's topic.
- **Action URL**: A pre-computed URL per Intelligence table row that, when clicked, opens the appropriate tool (Outlook, Asana, SharePoint, Slack) with context from the row pre-filled. Generated during the data refresh pipeline, not at click time.
- **Action button**: The clickable UI element on each Intelligence table row that opens the action URL. Replaces or supplements the existing dismiss button.
- **Command Center**: Dashboard home view at `~/shared/dashboards/index.html` with Hero, Daily Blocks, Integrity Ledger, Actionable Intelligence, and other sections.
- **command-center-data.json**: The JSON file generated by the refresh pipeline that feeds all Command Center sections, including the Actionable Intelligence table.
- **Outlook compose link**: A `mailto:` URL or Outlook deep-link that opens a new email draft with pre-filled recipient, subject, and optionally body text.
- **Slack deep-link**: A `slack://` protocol URL or `https://amzn-aws.slack.com/` web URL that opens a specific channel or DM in Slack.
- **Asana task creation link**: A URL that opens Asana's "new task" view with pre-filled fields (name, description, assignee, project) using Asana's URL scheme or API.
- **SharePoint artifact link**: A URL pointing to a specific document in SharePoint OneDrive (`Kiro-Drive/`) — typically a state file, wiki article, or strategic document relevant to the Differentiate item.
- **Five Levels**: Richard's sequential strategic priorities — L1 Sharpen Yourself → L2 Drive WW Testing → L3 Team Automation → L4 Zero-Click Future → L5 Agentic Orchestration
- **Soul principles**: The 6 "How I Build" principles in `soul.md` — Routine as liberation, Structural over cosmetic, Subtraction before addition, Protect the habit loop, Invisible over visible, Reduce decisions not options

## Scope

### In scope

- Per-type action URL generation logic in the data refresh pipeline (the code that produces `command-center-data.json`)
- Communicate items: generate Outlook compose link (`mailto:` with pre-filled recipient, subject, body snippet) or Slack deep-link (channel/DM URL) based on item metadata
- Delegate items: generate Asana task creation link with pre-filled task name, description (from item context), and assignee (if known)
- Differentiate items: generate SharePoint artifact link pointing to the relevant document, state file, or wiki article
- One "action" button per Intelligence table row in the Command Center UI
- Fallback handling when action URL cannot be generated (missing metadata)
- Action URL field added to the Intelligence table data model in `command-center-data.json`

### Out of scope

- Executing the action automatically (no auto-sending emails, no auto-creating tasks) — the link opens the tool; Richard completes the action
- Modifying how Intelligence items are generated or categorized — this spec only adds action links to existing items
- Adding new Intelligence item types beyond Communicate / Delegate / Differentiate
- Multi-step workflows (e.g., "create Asana task AND send Slack notification") — one action per row
- Tracking whether Richard clicked the action link or completed the action — no analytics or completion tracking
- Changing the dismiss button behavior — the action button is additive, not a replacement for dismiss
- Mobile or responsive layout — the dashboard is desktop-only
- Authentication flows for Outlook, Asana, or SharePoint — assumes existing credentials are available via MCP servers or browser session

## Requirements

### Requirement 1: Action URL Generation per Item Type

**User Story**: As Richard viewing the Actionable Intelligence table, I want each row to have a pre-computed action URL that opens the right tool for that item type, so I don't have to figure out where to go or manually transfer context.

#### Acceptance Criteria

1. WHEN the data refresh pipeline generates Intelligence table data, THEN each item SHALL include an `action_url` field containing a URL appropriate to its type (Communicate, Delegate, or Differentiate)
2. WHEN a Communicate item has a target person and topic, THEN the `action_url` SHALL be an Outlook compose link (`mailto:` with `to`, `subject`, and optionally `body` parameters) pre-filled with the recipient's email and a subject line derived from the item context
3. WHEN a Communicate item's preferred channel is Slack (indicated by item metadata), THEN the `action_url` SHALL be a Slack deep-link (`https://amzn-aws.slack.com/` web URL or `slack://` protocol URL) pointing to the relevant channel or DM
4. WHEN a Delegate item has a task description and optionally an assignee, THEN the `action_url` SHALL be an Asana task creation link that pre-fills the task name (from item title), description (from item context), and assignee (if known from item metadata)
5. WHEN a Differentiate item references a specific artifact, document, or data surface, THEN the `action_url` SHALL be a direct link to that resource — a SharePoint document URL, a wiki article URL, or a dashboard view URL as appropriate
6. WHEN an item lacks sufficient metadata to generate a meaningful action URL (e.g., no recipient for a Communicate item, no artifact reference for a Differentiate item), THEN the `action_url` SHALL be `null` and the action button SHALL NOT appear for that row

### Requirement 2: Communicate Item URL Generation

**User Story**: As Richard seeing a "Communicate" item (e.g., "Email Brandon about AU test results"), I want to click one button and have Outlook open with the recipient, subject, and a starter body already filled in, so sending the message is a 30-second edit instead of a 3-minute composition from scratch.

#### Acceptance Criteria

1. WHEN generating a Communicate action URL for email, THEN the system SHALL produce a `mailto:` link with: `to` set to the target person's email address (resolved from item metadata or `memory.md` relationship data), `subject` set to a concise subject line derived from the item title/context, and `body` set to a 1-2 sentence starter text summarizing the item's context
2. WHEN generating a Communicate action URL for Slack, THEN the system SHALL produce a Slack web URL (`https://amzn-aws.slack.com/archives/{channel_id}`) or DM URL that opens the relevant conversation
3. WHEN the item metadata includes both an email recipient and a Slack channel, THEN the system SHALL default to the Outlook compose link — email is the higher-friction action that benefits more from pre-filling (Slack messages are already low-friction)
4. WHEN the target person's email address cannot be resolved from item metadata, THEN the `action_url` SHALL be `null` for that item — do not generate a `mailto:` link with an empty `to` field

### Requirement 3: Delegate Item URL Generation

**User Story**: As Richard seeing a "Delegate" item (e.g., "Create task for MX market analyst to update state file"), I want to click one button and have Asana open with the task pre-filled, so delegation is a 10-second review-and-submit instead of a 2-minute task creation from memory.

#### Acceptance Criteria

1. WHEN generating a Delegate action URL, THEN the system SHALL produce an Asana task creation URL or deep-link that pre-fills: task name (from item title), task description (from item context — the "why" and "what" of the delegation), and project (the default Asana project for Richard's tasks, if determinable)
2. WHEN the item metadata includes an assignee (the person to delegate to), THEN the Asana link SHALL pre-fill the assignee field with that person's Asana identifier (email or user GID)
3. WHEN the item metadata does not include an assignee, THEN the Asana link SHALL leave the assignee field empty — Richard will fill it manually
4. WHEN the Asana task creation URL format does not support all desired pre-fill fields, THEN the system SHALL pre-fill what the URL scheme supports and document the limitation — partial pre-fill is better than no pre-fill

### Requirement 4: Differentiate Item URL Generation

**User Story**: As Richard seeing a "Differentiate" item (e.g., "Review AU state file for testing gaps"), I want to click one button and land directly on the relevant document, so I don't have to navigate SharePoint or the wiki to find it.

#### Acceptance Criteria

1. WHEN generating a Differentiate action URL, THEN the system SHALL produce a direct URL to the referenced artifact — a SharePoint document URL (for state files, published articles, or Excel dashboards), a local dashboard URL (for dashboard views), or a wiki article URL
2. WHEN the item references a market state file, THEN the `action_url` SHALL point to the SharePoint state file for that market in `Kiro-Drive/state-files/` (e.g., `https://amazon.sharepoint.com/.../{market}-state.md`)
3. WHEN the item references a wiki article or strategic document, THEN the `action_url` SHALL point to the published version in SharePoint `Artifacts/` or the local wiki file in `~/shared/wiki/agent-created/`
4. WHEN the item references a general topic without a specific artifact, THEN the `action_url` SHALL point to the most relevant existing surface (e.g., the Performance dashboard view for performance-related items, the relevant state file for market-specific items) — best-effort matching, not exact
5. WHEN no relevant artifact or document can be identified for a Differentiate item, THEN the `action_url` SHALL be `null` — do not generate a link to a generic or unhelpful destination

### Requirement 5: Action Button UI in Intelligence Table

**User Story**: As Richard scanning the Intelligence table, I want a single, clear action button per row that I can click to open the right tool, without cluttering the table or changing its existing layout.

#### Acceptance Criteria

1. WHEN an Intelligence table row has a non-null `action_url`, THEN the row SHALL display an action button (e.g., labeled "Act" or showing an action icon) that opens the `action_url` in a new browser tab when clicked
2. WHEN an Intelligence table row has a null `action_url`, THEN the row SHALL NOT display an action button — the row appears as it does today (with dismiss only)
3. WHEN the action button is displayed, THEN it SHALL appear alongside the existing dismiss button — not replacing it. The layout should be: [action content] ... [Act button] [Dismiss button]
4. WHEN the action button is displayed, THEN it SHALL use a visual style consistent with the existing Command Center button patterns (same font, padding, border-radius) but with a distinct color or icon that differentiates it from dismiss
5. WHEN the action button is clicked, THEN the URL SHALL open in a new tab (`target="_blank"`) — the dashboard stays open so Richard can continue processing other items

### Requirement 6: Data Model Extension

**User Story**: As the data refresh pipeline, I need the Intelligence table data model to include action URL metadata, so the dashboard can render action buttons without computing URLs at render time.

#### Acceptance Criteria

1. WHEN `command-center-data.json` is generated, THEN each Intelligence table item SHALL include an `action_url` field (string or null) and an `action_type` field (one of: `email`, `slack`, `asana`, `sharepoint`, `dashboard`, or null)
2. WHEN `command-center-data.json` is generated, THEN each Intelligence table item SHALL include an `action_label` field (string or null) containing a short human-readable label for the action (e.g., "Email Brandon", "Create task", "Open AU state file") — this label appears as the button tooltip or accessible text
3. WHEN the data model is extended, THEN existing fields (item type, title, context, dismiss state) SHALL remain unchanged — the action fields are additive
4. WHEN the data refresh pipeline cannot determine the action URL for an item, THEN all three action fields (`action_url`, `action_type`, `action_label`) SHALL be null — partial population (URL without label, or label without URL) is not allowed

## Design Constraints

1. **No new infrastructure**: Action URL generation must happen within the existing data refresh pipeline that produces `command-center-data.json`. No new services, APIs, or backend processes.
2. **No automatic execution**: Clicking the action button opens a tool with pre-filled context — it does NOT send the email, create the task, or modify the document. Richard always has the final say.
3. **Single-user assumption**: Richard is the sole user. No multi-user routing, no permission checks beyond existing SharePoint/Asana/Outlook credentials.
4. **URL scheme limitations**: `mailto:` links have body length limits (~2000 characters total URL). Asana's URL-based task creation may not support all fields. The design must document these limitations and degrade gracefully (partial pre-fill over no pre-fill).
5. **Existing pattern reuse**: The action button must visually integrate with the existing Intelligence table layout. Don't redesign the table — add one button per row.
6. **Offline resilience**: Action URLs are pre-computed during data refresh, not at click time. If the dashboard is loaded from a cached `command-center-data.json`, action buttons still work (they're just URLs in the data).
7. **Subtraction check**: This spec adds one button and one data field per Intelligence table row. It does not add new sections, new views, or new data sources. The net complexity increase is minimal: URL generation logic in the refresh pipeline + one button in the table renderer. If the action links prove unhelpful in practice, they can be removed by deleting the `action_url` field and the button renderer — no architectural changes needed.
